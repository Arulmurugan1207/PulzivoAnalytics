Owner Exclusion

Prevent your own activity from polluting your analytics data. Choose the method that fits your workflow.

How it works

The owner exclusion flag is per-browser, in-memory. Each visitor's browser runs an independent copy of the SDK — enabling it for yourself never affects other users' tracking.

Method 1 — One-time console command Easiest

Open your browser DevTools console on your site and run this once. It persists across all page refreshes automatically.

{{ codeBlocks['owner-console'] }}

Method 2 — After login (role-based)

Check the authenticated user's role after login and call setOwner(). This is the most reliable approach for apps with user authentication.

{{ codeBlocks['owner-login'] }}

Method 3 — At init time

Pass excludeOwner: true in your init call. Useful when you already know at page-load time (e.g. a separate admin domain).

{{ codeBlocks['owner-init'] }}

Method 4 — localStorage flag

Set pulz_is_owner in localStorage manually. The SDK reads it automatically on every page load — no code changes needed in your app.

{{ codeBlocks['owner-localstorage'] }}

Method 5 — Environment / hostname check

Automatically suppress tracking on localhost or staging environments so development activity never reaches your dashboard.

{{ codeBlocks['owner-env'] }}

Important Notes

  • The flag is in-memory — it resets on page refresh unless you use persist: true, disableTracking(), or the localStorage methods.
  • Suppression is silent — no errors, no console warnings (unless debug mode is on).
  • All event types are suppressed: page views, clicks, custom events, everything.
  • Call enableTracking() or setOwner(false) to undo at any time.
